# BSR2 Design Notes

## Identifier

```text
BSR2-ARX-SPONGE-ETM
```

## Envelope Version

```text
2
```

This document describes the current implementation of BSR2. It is not a security proof, standard, certification, or recommendation for production use.

---

## Construction Overview

The implementation combines:

- A custom 1024-bit permutation
- Sponge-style hashing
- Custom keyed authentication
- Purpose-separated subkey derivation
- A deterministic random-bit generator (DRBG)
- Counter-based stream generation
- An encrypt-then-authenticate envelope

Major functions use explicit domain strings to separate purposes.

---

BSR2 fixed parameters are reproducibly generated by:

```text
tools/generate_bsr2_parameters.py
```

The generator produces:

- Round constants
- Structural mixing constants
- Rotation values
- Word permutation values

These generated values are embedded into:

```text
brisart_security_primitives.py
```

For details see:

```text
docs/BSR2_PARAMETER_GENERATION.md
```

Reproducible parameter generation documents provenance. It does not establish cryptographic security.

## Permutation

The custom permutation operates on:

- 16 state words
- 64 bits per word
- 1024 total state bits
- 32 default rounds

Round operations include:

- Modular addition
- XOR
- Fixed rotations
- Round constants
- Word reordering
- Nonlinear neighbor mixing

Current rotation values:

```text
34, 5, 38, 37, 31, 50, 59, 17
```

Current word permutation:

```text
6, 11, 12, 0, 5, 15, 2, 14, 3, 13, 4, 7, 10, 9, 8, 1
```

---

## Sponge-Style Hash

The sponge-style construction uses:

- A 32-byte rate
- Domain-dependent initial-state processing
- A padding marker of `0x80`
- Zero padding
- An 8-byte big-endian bit length field
- A default output length of 32 bytes

Supported output lengths:

```text
1-1024 bytes
```

---

## Keyed Authentication

The keyed authentication construction uses separate inner and outer hash domains.

The implementation:

1. Frames the key
2. Frames the message
3. Produces a 64-byte inner value
4. Hashes the framed key, framed inner value, and framed message

This is a custom construction and has not been established as equivalent to HMAC or another standardized MAC.

---

## Password-Derived Key Experiment

Inputs:

- UTF-8 password
- Caller-provided salt
- Iteration counter
- Internal evolving state

The implementation requires:

- Non-empty password
- At least 16 salt bytes
- At least 10,000 iterations

Default iteration count:

```text
120000
```

This function is not memory-hard and has not been validated as a production password-hardening construction.

---

## Subkey Derivation

Per-purpose subkeys are derived from:

- Master key
- Message salt
- Purpose label

The envelope derives separate keys for:

- Encryption
- Authentication

---

## Deterministic Random-Bit Generator

The DRBG requires:

- At least 64 seed bytes
- At least 16 personalization bytes
- Non-empty additional input

The DRBG maintains:

- Mutable internal state
- Request counters
- Byte limits
- Continuous repeated-block checks
- Reseeding support
- Destroyed state handling

The DRBG expands caller-provided seed material.

The DRBG does not create entropy.

A valid reseed can reactivate a destroyed generator.

---

## Generator Restart And Reuse Requirement

The DRBG is deterministic.

Reinitializing the DRBG with the same:

- Seed
- Personalization
- Additional inputs
- Call sequence

will reproduce the same output sequence.

With the same master key and context this can reproduce:

- Message salt
- Nonce
- Derived working keys
- Stream output
- Ciphertext
- Authentication tag

Applications must:

- Supply fresh unpredictable seed material
- Or safely persist DRBG state across restarts

---

## Stream Generation

The stream generator repeatedly authenticates:

- A framed nonce
- A big-endian counter

under the encryption key.

Generated blocks are concatenated until the requested length is reached.

Plaintext is transformed using XOR with the generated stream.

---

## Authenticated Envelope

An envelope contains exactly:

```text
algorithm
version
salt
nonce
ciphertext
tag
```

Binary values are encoded using canonical lowercase hexadecimal strings.

### Encryption

The caller supplies:

- Master key
- Plaintext
- Context string
- Generator

The implementation:

1. Requests a 32-byte salt
2. Requests a 32-byte nonce
3. Verifies the generator returned bytes
4. Verifies salt and nonce lengths
5. Verifies salt and nonce are different
6. Derives encryption and authentication keys
7. Encrypts using the generated stream
8. Authenticates algorithm, version, context, salt, nonce, and ciphertext

### Decryption

The implementation validates:

- Envelope field set
- Algorithm identifier
- Envelope version
- Encodings
- Field lengths
- Ciphertext size

Authentication is verified before plaintext is returned.

Only authenticated ciphertext is decrypted.

---

## Size Limits

Current limits:

- Plaintext: 16 MiB
- Ciphertext: 16 MiB
- DRBG request: 1 MiB
- DRBG output before reseed: 16 MiB
- DRBG requests before reseed: 100,000

---

## Comparison Routine Limitation

The comparison routine avoids intentional early returns.

However, Python interpreter behavior has not been established as constant-time.

BSR2 does not claim verified side-channel resistance.

---

## Compatibility

BSR2 is intentionally incompatible with:

- BSR1 ciphertext
- BSR1 vectors
- BSR1 domains
- BSR1 fixed parameters

The historical BSR1 archive should be retained separately if BSR1 material must be inspected or recovered.

---

## Unestablished Properties

The implementation has not established:

- Collision resistance
- Preimage resistance
- Differential resistance
- Rotational resistance
- Algebraic resistance
- Related-key resistance
- Stream indistinguishability
- Authentication-forgery resistance
- Password-guessing resistance
- Entropy-source validity
- State-compromise resistance
- Side-channel resistance
- Production suitability
